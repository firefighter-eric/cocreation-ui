# 共创故事 App 产品设计文档

## 1. 产品定位

`cocreation-ui` 是一个面向中文短句故事共创的前端应用。
产品核心不是通用聊天，而是围绕“启动语句 + 轮流续写 + 参数可调 + 可导出结果”的单任务创作体验。

应用采用接近 ChatGPT 的白色双栏界面：

- 左侧为全局设置区
- 右侧为故事工作区
- 主交互围绕故事消息流和底部操作区展开

## 2. 产品目标

产品当前阶段目标：

- 让用户快速开始一段人机共创故事
- 支持机器自动生成对话样例，用于测试 prompt 和启动语句
- 让导出结果同时适合人工阅读和程序分析
- 让后续功能变更可以基于文档而不是口头约定推进

非目标：

- 不做通用问答助手
- 不做多人实时协作
- 不做账号体系和云端同步
- 不做复杂项目管理或工作流能力

## 3. 核心概念

### 3.1 启动语句

每段故事都从一个固定的启动语句开始。
启动语句由预设 seed 提供，包含：

- `id`
- `title`
- `openingLine`
- `summary`

当前 UI 中，左侧开场句列表每项只展示一行 `openingLine`，不展示额外说明。

启动语句不在聊天区由用户输入，而是由系统预先设定，并具有两层语义：

- 请求给模型时，作为第一条 `user` message
- 导出 JSON 时，作为 `conversation` 的第一项保留

### 3.2 创作风格

当前支持两种风格：

- `creative`：脑洞跳脱
- `coherent`：自然连贯

风格通过默认模板影响完整 `system prompt`。
当前产品不是隐藏式风格注入，而是：

- 设置抽屉中展示完整可编辑的 `system prompt`
- 切换风格时，直接改写这段可见文本
- 用户可在此基础上继续手动修改
- 默认模板语气应接近日常口语，像普通大学生讲故事
- 默认模板不使用书面化表达或生僻词要求

### 3.3 故事规则

默认故事规则：

- 单句最大长度 `20`
- 不允许标点
- 不允许空输入

这些规则当前对三种模式都生效。

## 4. 使用模式

### 4.1 与AI对话

这是默认模式。
用户手动输入一句，模型基于当前上下文续写一句，双方交替推进故事。

交互要求：

- 手动模式必须先点击“开始”按钮，之后才允许输入
- 按回车直接发送
- `Shift + Enter` 换行
- 中文输入法处于选词 composition 状态时，回车不能触发发送
- 生成中禁止重复发送
- 输入不符合规则时，发送按钮应直接禁用，并提供显眼的就地错误提示
- 模型请求超时后，需要退出生成中状态并允许用户重试
- 错误出现时在消息区展示错误提示
- 与AI对话模式需要记录关键行为数据，供导出 JSON 使用

手动模式当前需要记录这些信息：

- session 开始时间
- AI 每轮生成开始时间
- AI 每轮生成结束时间
- 用户每轮开始输入时间
- 用户每轮提交时间
- 用户每轮 `Backspace` 次数
- 用户每轮反应时

这些记录仅要求在与AI对话模式下采集。
与人对话模式沿用同一套底层时间记录。
AI自动对话模式不需要伪造同类输入行为数据。

### 4.2 与人对话

该模式在前台按真人对话呈现，用户应明确感知自己正在与另一位对话者共同续写故事。
界面结构与手动模式一致，用户同样需要先点击“开始”再输入。

交互要求：

- 消息区不显示 `AI` 身份，而使用“对方”语义
- 消息头像文案使用直接称呼：用户侧为“我”，对方侧为“对方”
- 前台文案不得出现“像真人”“模拟”“延迟”“模型生成”等破坏真人对话感知的提示
- 回复采用整句延迟显示，不做逐字流式
- 延迟由基础等待、按字数增加的等待和随机波动组成
- 延迟期间显示“对方正在输入”
- 导出 JSON 时，顶层 `mode` 必须是 `human_like`
- 导出仍保留真实生成时间字段，不隐藏 `ai_started_at`、`ai_ended_at`

### 4.3 AI自动对话

该模式用于自动生成完整样例，主要服务于 prompt、seed 和产品效果验证。

交互要求：

- 主区域展示自动对话操作区
- 自动模式使用设置抽屉中的最大回合数量
- 最大回合数量范围为 `1-10`
- 每 `1` 回合代表 `1` 组“用户一句 + AI 一句”
- 点击“自动生成 1 例对话”后，由系统连续生成整段对话

自动模式专属控件不放在左侧栏，只出现在主区域内。

## 5. 信息架构

### 5.1 左侧栏

左侧栏只承载全局设置：

- 模式选择
- 启动语句选择
- 重新开始

左侧栏支持手动收起与展开。
收起后右侧工作区应扩展占满可用宽度，并保留明显的展开入口。

左侧栏不承载自动模式专属参数。
左侧栏不承载创作风格和模型参数。

### 5.2 右侧工作区

右侧工作区包括：

- 顶部故事标题区
- 标题区右侧的当前规则信息
- 中间消息流
- 底部操作区
- 右上角设置按钮唤起的 prompt 设置抽屉

手动模式底部操作区只保留输入和开始相关控件。
自动模式底部操作区只保留自动生成按钮和当前回合说明。

两个模式的导出入口位置必须一致，统一放在右侧工作区右上角。
手动模式在开始之前，底部操作区需要展示“开始”入口并禁用输入框。
消息头像文案规则固定为：

- 与AI对话：`我 / AI`
- 与人对话：`我 / 对方`

## 6. 导出设计

当前导出目标是保留完整上下文和元信息，供后续分析、回放和程序处理。

### 6.1 文件命名

导出时仅产出一个文件：

- `cocreation-yymmdd-hhmmss.json`

时间使用用户本地时间，字段均补零。

### 6.2 JSON 内容

JSON 用于保存完整上下文和元信息，至少包含：

- `session_id`
- `session_started_at`
- `system_prompt`
- `model_settings`
- `exported_at`
- `mode`
- `style`
- `status`
- `error`
- `rules`
- `seed`
- `conversation`

其中 `conversation` 需包含启动语句和全部消息。
普通消息项需保留：

- `id`
- `role`
- `content`
- `created_at`
- `is_opening`

其中手动模式消息还需按角色补充交互记录：

- 用户消息记录 `input_started_at`、`input_ended_at`、`backspace_count`
- 用户消息记录 `reaction_time_ms`，并补充 `reaction_reference_at`
- AI 消息记录 `ai_started_at`、`ai_ended_at`

用户反应时规则：

- 如果这是用户首轮输入，则从 `session_started_at` 计算到 `input_started_at`
- 如果这不是用户首轮输入，则从上一条 AI 的 `ai_ended_at` 计算到 `input_started_at`

JSON 导出字段统一使用 snake_case 命名，不使用 camelCase。
更完整的字段说明见 [export-schema.md](/Users/eric/projects/cocreation-ui/docs/export-schema.md)。

开场句在导出中使用：

- `role = user`
- `is_opening = true`

后续普通消息使用：

- `role = user | assistant`
- `is_opening = false`

### 6.3 Prompt 设置

右上角提供一个“设置”按钮，点击后从右侧弹出设置抽屉。

当前抽屉至少提供：

- 创作风格选择
- 一个可编辑的 system prompt 文本框
- 最大回合数量输入
- 开始回合设置
- `temperature`、`top_p` 等模型参数输入
- `model` 输入与候选模型选择

最大回合数量规则：

- 所有模式都可在设置抽屉中统一设置最大回合数量
- 默认值为 `5`
- 取值范围为 `1-10`
- `1` 回合表示 `1` 组“用户一句 + 对方一句”
- 与AI对话和与人对话达到最大回合后，应停止继续输入并提示已到达上限
- AI自动对话模式使用这项全局设置生成完整对话，不再在主区域单独设置轮数

开始回合规则：

- 所有模式都可在设置抽屉中统一设置开始回合
- 可选值为“用户”“对方”“随机”
- 默认值为“随机”
- “随机”应在每次新会话开始时重新决定本次实际起手方
- 与AI对话和与人对话中，如果本次由“对方”先开始，则点击“开始”后应先生成对方一句，再允许用户输入
- 与人对话模式前台仍保持“对方”语义，不显示 AI 身份
- `base URL`
- `API key`
- 保存按钮
- 清空按钮

当前产品约束：

- 设置里的文本就是完整最终 system prompt
- 风格切换会直接改写可见的 system prompt 文本
- 开场句不写入 system prompt，而是作为请求中的第一条 user message
- 默认规则仍然保留，不允许被 UI 层删除
- 默认模型参数为 `temperature = 1.5`、`top_p = 1.0`
- 默认模型名取环境变量中的 `VITE_LLM_MODEL`，未配置时使用默认值
- 用户可在设置抽屉里填写自己的 `base URL` 和 `API key`
- 这份 API 配置保存在浏览器本地 `localStorage`，用于避免反复填写
- 设置抽屉中的 `model` 也保存在浏览器本地，避免重复填写
- 若用户配置完整，则优先覆盖环境变量
- 若用户配置不完整，则忽略该配置并回退到环境变量
- 若用户配置和环境变量都不完整，则回退到 Mock provider
- 用户可点击“获取候选模型”从当前 `base URL` 拉取模型列表
- 候选模型拉取失败时，不阻塞手动填写 `model`
- `API key` 输入框支持点击眼睛按钮切换显示与隐藏
- 保存后影响后续回复
- 导出 JSON 时需要写入 `system_prompt`
- 导出 JSON 时需要写入当前模型参数
- 导出 JSON 时不写入 `base URL` 和 `API key`

### 6.4 请求消息组织

当前发给模型的 `messages` 组织方式为：

1. 第一条 `system`：设置抽屉中当前可见的完整 `system prompt`
2. 第二条 `user`：当前 seed 的 `openingLine`
3. 第三条开始：真实对话历史，按 `user/assistant` 顺序追加

因此当前产品约束是：

- 开场句不影响 system prompt 文本本身
- 开场句属于请求上下文的一部分
- 导出中的开场句与请求格式保持一致，使用 `role = user`

## 7. 技术与架构约束

### 7.1 分层原则

产品逻辑必须保持分层，避免把规则和接口细节写进组件：

- `entities`：类型和纯状态机
- `features`：UI 组件和交互组织
- `shared/config`：产品默认配置
- `shared/lib/llm`：模型 provider 和 prompt 策略
- `shared/lib/validation`：输入输出规则
- `shared/lib/export`：导出逻辑
- `shared/lib/storage`：存储抽象

### 7.2 Provider 约束

前端只依赖统一 provider 接口，不直接在组件中 `fetch`。
当前默认支持：

- OpenAI-compatible provider
- Mock provider

运行时 provider 配置来源优先级为：

1. 设置抽屉中用户保存的完整 API 配置
2. 环境变量中的默认 API 配置
3. Mock provider

### 7.3 状态约束

故事会话由统一 session state 驱动。
至少覆盖这些状态：

- `idle`
- `ready`
- `submitting_user_line`
- `waiting_for_ai`
- `ai_replied`
- `failed`

## 8. 后续修改规则

后续如果调整产品，应优先修改本文件，再修改代码。
新需求应明确说明变更的是哪一类内容：

- 产品目标
- 交互规则
- 导出格式
- 模式定义
- 数据结构
- UI 结构

若新需求与本文档冲突，默认以最新文档为准，并同步更新 README 中的说明。

## 9. 当前验收标准

当前版本应满足：

- 可以在与AI对话模式完成多轮接龙
- 可以在与人对话模式下按真人对话语义完成多轮接龙
- 可以在AI自动对话模式按 `1-10` 轮生成样例
- 三种模式都可以导出
- 导出时仅下载 JSON
- JSON 包含完整元信息和消息明细
- 与AI对话模式的 JSON 包含用户输入与 AI 生成的时间记录
- 设置抽屉可以直接修改完整 system prompt、创作风格和模型参数
- 设置抽屉可以填写并持久化 `base URL` 与 `API key`
- 设置抽屉可以拉取候选模型，并保存最后选择的 `model`
- 请求中的开场句作为第一条 `user` message
- 文件名统一为 `cocreation-yymmdd-hhmmss.json`
